Why RAG Works in Demos but Fails in Production

The demo corpus is clean, small, and static. Production is messy, growing, and adversarial. RAG does not "break" — your assumptions about the corpus do.

If you are asking why RAG fails in production, you already passed the easy part. The hard part is that failures are gradual: accuracy erodes over weeks until a customer email makes it undeniable.

Failure mode 1: Stale or partial corpus

Demos index ten perfect PDFs. Production adds hundreds of formats — scans, tables, appendices, bilingual sections. Chunks miss the sentence that mattered. The model fills gaps convincingly. Users call it hallucination; ops calls it ingestion debt.

Failure mode 2: Query distribution shift

Launch week queries match your eval set. Month three: edge cases, compound questions, users testing limits. Retrieval tuned for short FAQs fails on multi-hop questions unless you redesign chunking and routing.

Failure mode 3: No production eval loop

Offline benchmarks green; no sampled human review, no thumbs signal, no regression on document updates. Production RAG reliability requires continuous eval — not a one-time launch checklist.

Failure mode 4: Context window as a crutch

When retrieval misses, teams inject more chunks or paste entire documents. Latency and cost spike; signal-to-noise drops. Bigger windows mask bad retrieval instead of fixing it.

Failure mode 5: Agent loops amplify errors

Agent retrieves wrong chunk → model confident → tool call doubles down → expensive wrong answer. Agentic RAG needs circuit breakers: max steps, confidence gates, human escalation paths.

What to do first

Pick your worst production failure from support tickets. Trace: ingestion version, retrieved chunks, model, outcome. Fix that path before re-architecting the vector DB. See production RAG best practices for the operating checklist.
