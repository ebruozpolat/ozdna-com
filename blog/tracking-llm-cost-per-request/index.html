Tracking LLM Cost Per Request, User, and Workflow

Provider dashboards show aggregate spend. Product teams need attribution: which request, which user, which job — so pricing and routing decisions are grounded in data.

LLM cost per request tracking is the foundation of token cost management. Without it, credits on your pricing page are fiction and routing experiments are blind.

Level 1: Per request

Every inference event logs: request_id, model, tokens_in, tokens_out, latency_ms, status. Attach parent trace_id when one user action fans out to retrieve + classify + generate. This is minimum viable observability — gateways often give you this layer.

Level 2: Per user / account

Roll up by user_id for B2C, account_id for B2B. You will find power-law distribution: top 5% of accounts drive disproportionate burn. That insight drives fair-use policies, sales qualification, and enterprise pricing — not angry surprises at renewal.

Level 3: Per workflow

The layer gateways miss. Map events to workflow_id — the job the user thinks they bought. Example workflows: compliance_delta_report, thesis_ai_scan, support_ticket_draft. Margin lives here. If one workflow is underwater, fix routing for that path only.

From logs to decisions

  1. Weekly: top 10 workflows by spend and by spend-per-successful-outcome.
  2. Monthly: cohort margin — new users vs. retained, plan tier vs. actual burn.
  3. Per release: did prompt or model change shift tokens/outcome?

Feed results into token economics — credits should trace back to these rollups.
