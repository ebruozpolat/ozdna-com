Production RAG Best Practices: What Breaks After Launch

Demo RAG is a notebook and a PDF. Production RAG is ingestion pipelines, stale documents, eval debt, and latency that spikes when retrieval fails twice and the model hallucinates confidently.

Production RAG best practices are not "pick Pinecone vs. Weaviate." They are operational habits that keep production RAG reliability from decaying after the launch party.

1. Treat ingestion as a product surface

Documents change. Regulations update. Course materials revise. If ingestion is a one-off script, retrieval quality drifts silently. Schedule syncs per source tier, version chunks, and log document_version on every answer. Users blame the model; the bug is stale vectors.

2. Eval per document type, not one golden set

Compliance PDFs, chat logs, and academic papers fail differently. Maintain eval suites by type with metrics that matter: citation accuracy, refusal rate on missing context, latency p95. Run on every ingestion or prompt change.

3. Retrieval governance beats bigger context windows

  • Source tiering — official gazette vs. commentary vs. user upload.
  • Confidence thresholds — abstain or escalate when similarity scores dip.
  • Chunk overlap discipline — redundant chunks inflate tokens without improving recall.
  • Human review queues for high-risk verticals (finance, law, health).

4. Observe cost per query, not just accuracy

Failed retrieval often triggers bigger models, wider search, or multiple re-tries. Track tokens per successful answer. A "smarter" pipeline that needs three retrieval passes can be worse for margin than a simpler one with higher precision.

5. Plan for failure modes explicitly

Document what happens when: corpus is empty, OCR garbles a scan, user language mismatches indexed language, provider embedding API is down. Production teams need runbooks, not optimistic system prompts.
